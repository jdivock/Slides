Jafar Husain: Async Programming in ES7

In which I attempt to explain generators and async in 5 minutes

Ship every year

  • ES6 -> ES2015
  • ES7 -> ES2016

Better and easier Async functions

Callback hell sucks

Even with promises we need to forward errors up the chain

Generators produce data via iteration


// data producer
function* nums() {
  yield 1;
  yield 2;
  yield 3;
}

for...of loop consumes via iteration


// data consumer
function printData() {
  for(var x of nums()) {
    console.log(x);
  }
}

That's great for synchronous data

Not so much for async data

So how about for..on instead


// data producer
async function* nums() {
  yield 1;
  yield 2;
  yield 3;
}

// data consumer
async function printData() {
  for(var x on nums()) {
    console.log(x);
  }
}

Mutations in Relay

Laney Kuenzel

Neither React nor Flux solve data fetching

Need to prevent scenarios where you're over-fetching

Or worse, under-fetching and your page explodes

The current react structure has implementation details of components leaking to their parents

Changing a child neccessitates modiying all relevant parents

But what if we could put the data query in the component

I showed something similar during my parse presentation

It's hard to reason about when your render logic is on your client and your data fetching logic is on the server

What fields are getting sent down?

Of that data what are we using in our application?

So how should this work in a perfect world?

Put the data fetching logic in the component and have it be totally self contained

GraphQL

Data query language

FB's client apps have been using it to describe their data needs for the past couple years

Query only what you need

Composable, can build queries from other queries

Relay

Takes the queries from each compnent in the tree and builds an overall query and sends that to the server

Response is put in a single store

One store helps with data consistency and lessens boilerplate common with Flux

Mutations

Prior to relay mutation custom endpoints were set up

Didn't scale well

Mutation requirements

  • Operation you want to perform
  • Required inputs
  • Query for the data you need post-operation

Common scenarios built in

  • Optimistic updates
  • Error handling
  • Race conditions
